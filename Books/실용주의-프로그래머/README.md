- 객체 지향 프로그래밍에서 자주 볼 수 있는 연쇄적 호출
  
  ```java
  method()
      .find()
      .calculate()
      .aggregate()
  ```
  
  보다는..
  
  ```java
  a = method();
  b = find(a);
  c = caculate(b);
  d = aggregate(c);
  ```
  
  와 같은 방법이 프로그래밍의 변환에 적합하며, 코드내 결합도를 낮추기 적절하다.

#### Topic 41. 테스트로 코딩하기

- 테스트의 목적은 버그찾기가 아니라 테스트 그 자체에 있다.
  
  - 코드 작성을 하며 테스트에 대해 생각하면 코드를 어떻게 작성할지 알 수 있다는 것이 테스트의 가장 큰 장점

- 테스트 주도 개발(TDD, test-driven development), 단위 테스트(unit test), 임시 테스트(ad-hoc test)

- 테스트 먼저, 코드와 테스트를 함께, 나중에 테스트. 여러 대안이 있다.
  
  - 이중 "나중에 테스트"는 순 거짓말이다. 테스트를 안하겠다는 말과 동일하다.

### 8장. 프로젝트 전에

#### Topic 45. 요구사항의 구렁텅이

> 자신이 뭘 원하는지 정확히 아는 사람은 없다.

- 우리 개발자는 보통 고객의 **요구사항**에 부합하는 제품을 구현한다.
  
  - 이때 고객의 요구 사항은 추상적인 부분이 많다.

- 짧은 주기의 반복적인 피드백으로 추상적인 요구사항을 확인해야 한다. 그래야 우린 궤도에서 벗어나지 않을 수 있다.

- 요구사항 명세서와 같은 경우는 고객을 위한게 아니다. 개발자를 위한 것이다.
  
  - 그렇다고 명세서를 너무 자세하고 장황하게 작성하지 말자. 좋은 요구사항은 추상적이다. 우린 그 추상에서 같은 구현을 보고 작업해야 한다.

- 정채과 요구사항을 구별하자.

#### Topic 48. 애자일의 핵심

> 애자일은 명사가 아니다. 애자일은 무언가를 하는 방식이다.

- 작가가 제안하는 애자일하게 일하는 방법
  
  1. 여러분이 어디에 있는지 알아내라.
  
  2. 도달하고 싶은 곳을 향하여 의미 있는 발걸음을 가능한 한 작게 옮겨라.
  
  3. 어디에 도착했는지 평가하고, 망가트린 것이 있으면 고쳐라
  
  부끄럼쟁이 코드를 작성하거나..테스트를 진행하거나..
  
  결합도를 낮춰 바로 고치기 쉬운 코드를 작성하거나..
  
  책의 내용과 "애자일"하게 일하는 방법의 접점이 많이 보인다.

## 9장. 실용주의 프로젝트

#### Topic 49. 실용주의 팀

- 팀 규모는 안정적인 숫자로
  
  - 팀원 수 n의 O(n\*\*2)만큼 소통수단이 증가한다.
  
  - 적절하게 안정적으로 운영할 수 있는 팀을 꾸려라

- DRY를 지키기 위해 서로에게 관심을 갖자
  
  - 서로가 서로에 대한 관심은 프로젝트가 궤도에 벗어나지 않게한다.

#### Topic 50. 코코넛만으로는 부족하다.

- 방법론과 기술의 특장점에 매몰되지 말라.

#### Topic 51. 실용주의 시작 도구

- 자동화를 잊지말자
  
  - 반복적인 수작업은 일관성을 해친다.
  
  - 반복적인 수작업에도 오류가 발생하지 않는다고? -> 더더욱 자동화할 필요가 있다.

- 테스트를 잊지말자
  
  - 테스트를 위한 테스트도 잊지말자.

> 시스템을 설계하는 방법은 두가지 있다. 너무 단순해서 명확한 결함이 없거나, 너무 복잡해서 명확한 결함이 없는것이다.
